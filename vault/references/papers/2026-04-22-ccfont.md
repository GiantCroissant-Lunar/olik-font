# CCFont — component-based Chinese font generation

**Summary (one line).** GAN-based font generator that decomposes each character into up to 17 components via CJKradlib, extracts local style per component, and recombines them in the image domain to synthesise a target-style glyph — including zero-shot generation for unseen characters and styles.

> MDPI blocked WebFetch (403). Content here is pieced together from the CCFont GitHub README (https://github.com/xjnpark/CCFont), the MDPI abstract surfaced via WebSearch, and ResearchGate figure captions for the CRC module (fig15). Anything I could not verify from those three sources is not stated.

## Ideas we could borrow

1. **Use CJKradlib's RadicalFinder as a second decomposition source** next to `cjk-decomp` (currently ingested in `project/py/src/olik_font/sources/cjk_decomp.py`). Cross-checking the two lists would surface decomposition disagreements and feed Plan 11's `source_stroke_indices` more reliably. *Cost: new adapter module under `sources/`; rough effort 0.5 day; prerequisite — wait for Plan 11 to land (Archon is editing `sources/`).*
2. **Per-component local style conditioning** — even without a neural net, the idea that style information is most accurate when localised to component geometry (not aggregated over a full glyph) matches our prototype-library direction. Concretely: when we later add a styling pass, it should consume per-`InstancePlacement` strokes, not the composed raster. *Cost: design note only right now; becomes real when `packages/glyph-styler/` exists.*
3. **Zero-shot evaluation harness.** CCFont's validation strategy — generate characters the model has never seen — is a useful template for our extraction: for any glyph not in our seed set, we should be able to compose from the prototype library alone. Adopt the "first-seen char" split as a standing QA metric. *Cost: a CLI subcommand + a held-out list; ~0.5 day; blocked on Plan 11 compose rewrite.*

## INCOMPATIBLE ideas (on record)

- **CCFont's composition step is image-space GAN recombination, not affine composition.** Adopting that recombination would replace measured per-instance transforms with a learned raster synthesiser. `INCOMPATIBLE: violates .agent/skills/05-engine/glyph-geometry/SKILL.md — do not adopt.`
- **"Up to 17 components" as a hard cap** — using a fixed component-count bucket to drive layout (rather than a measured decomposition tree) would be the categorical pattern the skill forbids. `INCOMPATIBLE: violates .agent/skills/05-engine/glyph-geometry/SKILL.md — do not adopt.` (Reading the decomposition as a count hint is fine; making compose depend on the bucket is not.)

## What it would cost us to adopt the compatible parts

- Module: new `project/py/src/olik_font/sources/cjkradlib.py` (adapter, mirrors the cjk_decomp pattern).
- Prerequisites: Plan 11 landed; `extraction_plan.yaml` consuming the new adapter can be gated behind a `decomp_source` field we already have (D10).
- Risk: CJKradlib is a runtime Python dep; pin version and vendor its data file to keep deterministic builds.
