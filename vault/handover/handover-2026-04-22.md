## Today's bug hunt — detailed

### Bug 1: `_proto_index_from_db` string-parsing of `source`

Plan 03's prototype emitter stores `source` as a structured dict:

```json
{"kind": "mmh-extract", "from_char": "明", "stroke_indices": [0, 1, 2, 3]}
```

The Plan 09 code in `batch.py::_proto_index_from_db` did:

```python
source = str(row.get("source", ""))
from_char = str(row.get("from_char", ""))
if not from_char and source:
    from_char = source[-1:]
```

`str({"kind": "mmh-extract", ...})` is `"{'kind': 'mmh-extract', ...}"` — ends in `}`. So `from_char = "}"`. When `extract_all_prototypes` later did `mmh[pp.from_char]`, every reused prototype raised `KeyError: '}'` and the glyph landed as `failed_extraction`.

Visible symptom: fresh DB with 4 seeds → run `olik extract auto --count 100` → 0 verified, ~85 failed_extraction. Confirmed via the seeded smoke (committed as `29b97ae` alongside the fix).

### Bug 2: Canonical prototype extraction from wrong source

Even after fix 1, the auto-planner minted canonicals by slicing the **context char's** strokes 50/50 between components (naive `_stroke_range_for_component`). So `proto:u6728` ("木") minted while processing 稀 received **稀's first 6 strokes**, which include 1 non-木 stroke. Later chars using 木 via this canonical got geometrically-wrong strokes.

**Fix**: canonicals now extract from the component's **own** standalone MMH entry. `_extract_canonical_prototype(component_name, proto_id, mmh)` looks up `mmh[component_name]` and uses all its strokes. Raises RuntimeError if the component has no standalone MMH entry — the planner converts to `PlanFailed`.

Also removed the `_SLUGS` friendly-name map (`木→tree`, `日→sun`, etc.). Auto-planned canonicals now use `proto:u<hex>` slugs exclusively. This **intentionally** does not reuse the 4 hand-tuned seed prototypes (`proto:sun`, `proto:moon`, `proto:tree`, `proto:water_3dots`, ...) because those were extracted from context chars rather than standalone sources — reusing them propagated the old bug. The seeds stay in the DB as-is for 明/清/國/森 rendering; auto-planned glyphs use parallel `proto:u<hex>` canonicals.

### Bug 3: Presets used y-down bbox semantics

`apply_top_bottom` set `top_bbox = (gx0, gy0, gx1, split_y)` — y=[0..501] for the top slot. That's y-down semantics (small y = top). But MMH stroke paths are y-up (y=0 bottom, y=1024 top) throughout the pipeline, and renderers apply a global `translate(0,h) scale(1,-1)` flip at display time.

Result: compose placed "top" component at y=[0..501]. After the render-time flip, y=[0..501] → visual y=[h-501..h] = **bottom half of SVG**. So the top component rendered at the bottom and vice versa. Same bug in `apply_repeat_triangle` (top-center cell was placed at low y).

**Fix**: swap to y-up bbox:

```python
# apply_top_bottom (new)
split_y = gy0 + height * (1.0 - weight_top)
top_bbox = (gx0, split_y + gap, gx1, gy1)       # HIGH y = visual top
bottom_bbox = (gx0, gy0, gx1, split_y)          # LOW y = visual bottom
```

and the equivalent swap in `apply_repeat_triangle` (top-center cell now at `gy1 - cell_h..gy1`, bottom cells at `gy0..gy0 + cell_h`).

**Pre-existing regression**: this bug affected the seed 清 too (which has a nested top_bottom inside its 青 component) but was never visually spot-checked via preview-glyph.py before today, only via the inspector's xyflow views which don't render the composed glyph. Yesterday's "pass-1-complete" verification was valid for the views it tested but didn't exercise preview-glyph's composed render on a top_bottom case. Plan 09's auto-extraction surfaced it because many random chars use top_bottom.

Tests that encoded the old (buggy) y-down expectation were updated:
- `tests/test_preset_top_bottom.py::test_top_gets_upper_portion`
- `tests/test_preset_repeat_triangle.py::test_three_instances_placed_at_triangle_vertices`
- `tests/test_compose_walk.py::test_senr_repeat_triangle_resolves`

### Bug 4: Variant extraction cannot work without MMH `matches`

**Discovered**: MMH's `matches` field (per-stroke component assignment) is null for all 9574 entries. Without it, we have no reliable way to isolate which strokes of a context char belong to which component. Minting a context variant by slicing context strokes 50/50 produces wrong geometry (same root cause as bug 2).

**Current stance (Plan 09.1)**: if canonical reuse fails the IoU gate, the planner returns `PlanFailed` instead of minting a wrong variant. The glyph lands in `failed_extraction`. Proper variant extraction with IoU-per-stroke matching is a **Plan 09.2** follow-up.

Visible effect: `variant_of` table stays empty. No variant edges in the current DB.

## Known limitations / follow-ups

### Status distribution is honest, low, and correct

A fresh 100-char batch (seed=7) now produces approximately:

- `verified` 1–5 (IoU ≥ 0.90)
- `needs_review` 60–65 (composition structurally correct, IoU 0.65–0.89 vs MMH reference)
- `unsupported_op` 15–25 (ops not in our LUT: `stl`, `st`, `sl`, `str`, `lock`, `me`, `w`, `wtl`, `d/t`, `d/m`, `sbl`, `wd`, `rrefl`, `ra`, …)
- `failed_extraction` 2–15 (MMH data gaps + post-fix "canonical IoU fails, no variant extraction available")

The 1% verified rate reflects that compose-with-standalone-components produces the **right shape** but not pixel-exact positioning vs MMH's pre-compressed reference rendering. The spec expected this and flagged `needs_review` as the main reviewer queue. Visually, `needs_review` chars at 0.75+ render correctly (components in right position, just slightly different proportions than MMH's hand-drawn reference).

### Plan 09.2 backlog

1. **IoU-per-stroke matching for variant extraction**: Measure IoU of each canonical stroke vs each MMH stroke of the context char; Hungarian-style assignment finds which strokes belong to the component. Then re-extract with those indices as the variant. Would lift verified rate substantially.
2. **Speculative canonical probe**: Before committing to reuse, actually compose the char with the canonical and measure IoU. If below gate, refuse reuse even if canonical exists (current `_trivial_probe_iou` always returns 1.0). Needs Bug 4's variant extraction to be useful.
3. **Operator LUT expansion**: 15% of chars currently land in `unsupported_op`. Top unmapped operators: `stl` (8-10 occurrences per 100), `st` (3-5), `str`, `lock`. Adding these compose modes is a Plan 12 candidate.
4. **`olik extract backfill-status` should also set `iou_mean`**: Currently only updates `status`. Pre-Plan-09 rows (the 4 seeds) had `iou_mean` already via Plan 08's emit; this is a no-op for them but worth documenting.

### Archon workflow `create_pr` prompt is unsafe

Plan 08 and Plan 09's workflows both had the `create_pr` step shell-interpolate backticks from markdown code blocks in the PR body. Plan 08 recovered benignly (it only re-edited the PR). Plan 09 did NOT recover — it executed `task extract:backfill` + `task extract:batch-100` + `task extract:batch-500` against the **host** SurrealDB before catching itself, adding 969 `failed_extraction` rows (wiped via `olik db reset`).

**Fix for Plan 10's workflow**: the `create_pr` prompt must instruct Codex to use `gh pr create --body-file <tmpfile>` rather than inline heredoc. The YAML template in `.archon/workflows/plan-09-bulk-extraction.yaml` still has the bad pattern — copy-rename, and patch before the next workflow runs.

### TS apps not re-verified against compose fix

The y-up bbox fix changes the coordinate shape of composed records. LEFT-RIGHT chars (明, 清 at top level, 肺, etc.) are unaffected because they span full y range. TOP-BOTTOM and REPEAT-TRIANGLE renders will change.

Inspector's Decomposition Explorer / Placement Debugger views use flow-nodes which renders each **prototype** individually (not the composed glyph), so they're unaffected. Quickview renders composed glyphs via glyph-svg.tsx with its own flip — should behave correctly for the same reason preview-glyph.py does. Remotion-studio compositions likewise.

**Post-resume check**: open the inspector's Placement Debugger on 清, confirm the sheng/月 nodes show in the correct top/bottom order in the tree. If the tree visualization is data-driven from the layout_tree (not geometry), it'll be fine.
