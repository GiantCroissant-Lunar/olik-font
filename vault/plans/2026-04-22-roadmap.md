# Olik-font versioned roadmap — revised

Previous draft split data extension + solver work across five small
versions. That was wrong. Real milestones are bigger: v1 is "the seed
works AND can be styled," v2 is "all of the Taiwan 4808 work AND can be
styled," v3 is "10000+ characters covered."

## Guiding rule (binding across every version)

`.agent/skills/05-engine/glyph-geometry/SKILL.md` — component placement
is a measured affine transform in 1024² virtual space. Never preset
vocabulary, never global slot weights, never categorical flags that
patch preset behaviour. If a decomposition source (cjk-decomp, MMH
`matches`, etc.) actively misleads placement for a given char, we
REPLACE or BYPASS that source — we do not reintroduce preset-style
compensations.

## v1 — Seed quality + ComfyUI styling (complete)

Measured geometry side (delivered by Plan 11 + follow-up commits):
- 4 seed chars 明 / 清 / 國 / 森 render MMH-faithful.
- 127 scaled-up Taiwan chars verified end-to-end (IoU=1.000 on sample).
- Preset vocabulary gone from code, data, and emitted JSON.
- `compose/walk.py` reads measured transforms; bulk planner drives
  placement from MMH `matches` partition; op-whitelist gate removed.

Styling bridge side (delivered by Plan 12):
- `olik_font.styling` ships a ComfyUI REST client, base-PNG renderer,
  and batch orchestrator exposed through `olik style`.
- `project/comfyui/workflows/` includes `ink-brush`,
  `aged-print`, and `soft-watercolor` workflow graphs plus a README
  documenting required checkpoints.
- `scripts/verdict_styling.py` runs the kimi CLI verdict sweep over
  `styled-output/<char>/<style>/<seed>.png` and enforces the visual
  acceptance gate.
- v1 acceptance is now met: 4 seed chars × 3 styles × 2 seeds can be
  rendered through the measured geometry pipeline and styled through
  ComfyUI without reintroducing preset placement logic.

## v2 — All 4808 Taiwan chars correct and stylable (the big one)

Scope:
- Every character in the MoE Taiwan 4808 list that our pipeline touches
  reaches `verified` status OR is explicitly skipped with a recorded
  reason. Fallback failure modes (class C / D / A from the earlier
  mapping) all get real fixes instead of "structural" dismissals.
- All verified chars are stylable through the same ComfyUI bridge
  v1 delivers, with per-batch style consistency.

Deliverables (merged into one version on purpose; these don't ship
independently):

- **Recursive decomposition.** Plan 13 Task A. Extend
  `bulk/planner.plan_char` so when cjk-decomp arity > MMH top-level
  partition arity, it descends into `bulk/mmh_partition.nested_partition`
  and keeps going until arity matches. Current class-A failures (29
  chars) resolve here.
- **animCJK graphics + dictionary ingestion.** Plan 13 Task B. New
  `sources/animcjk.py` loader parallel to `sources/makemeahanzi.py`.
  A unified `char_graphics_lookup(char)` tries MMH first, then
  animCJK. Covers class-C failures (host char missing from MMH).
- **Component carving from containing hosts.** Plan 13 Task C. When a
  named component (乏, 侖, ⺊, ㇐, …) has no standalone graphics entry
  in either source, locate it inside a parent char via cjk-decomp and
  carve its strokes through the partition chain. Covers class-D
  failures (28 chars).
- **Per-prototype variant cap config.** Plan 13 Task D. Replace the
  global `--max-variants-per-proto` with a per-prototype cap defaulting
  to 10. Closes class-E residual needs_review.
- **cjk-decomp replacement for proven-wrong entries.** Plan 13 Task E.
  Where cjk-decomp yields a partition that conflicts with MMH `matches`
  geometry for chars we've confirmed look correct in MMH, override with
  animCJK's `decomposition` IDS string or a small local `cjk-decomp-
  overrides.json`. Logged per-char in `vault/references/
  cjk-decomp-overrides.md` so the rationale is traceable.
- **Styling pipeline scales.** Plan 13 Task F. The ComfyUI bridge from
  Plan 12 runs in batch; produces style-consistent variants across
  every verified char for at least 1 style family.
- **Visual verdict sweep.** Plan 13 Task G. Kimi CLI verdict on a
  random 200-char sample covering geometry + styling.

Exit criteria:
- `olik extract report`: verified ≥ 4700 out of 4808 buckets filled
  (≥ 97.8% coverage).
- Remaining non-verified chars are individually enumerated with
  per-char reasons in `vault/references/v2-unresolved.md` (each entry
  is either "data truly unavailable" or "replaced with manual
  override").
- Every verified char has at least one styled variant produced through
  the ComfyUI bridge without manual intervention.
- Zero preset vocabulary residuals anywhere (same gate as Plan 11).

## v3 — 10000+ characters

Scope: expand beyond the Taiwan 4808 to a ~10k-char pool (Simplified
Chinese, Japanese kanji, Korean hanja, historical / rare CJK). Mechanism:

- Add Simplified-Chinese / Japanese / Korean adapters alongside animCJK
  (`graphicsZhHans.txt`, `graphicsJa.txt`, `graphicsKo.txt`).
- Import CHISE IDS as a third decomposition source when cjk-decomp +
  animCJK both miss or disagree.
- Per-language pool files mirroring the MoE-4808 pattern, with the
  same `filled / verified / needs_review / failed_extraction`
  bookkeeping.

Exit criteria: verified ≥ 9500 out of 10000 covered chars. Every
verified char styled through the v1 ComfyUI bridge.

## v4+ (deferred)

- Anchor-binding solver (original spec D6) — alternative authoring path
  for chars where stroke partition is ambiguous.
- Learned layout adapter (IF-Font / CCFont-style) — trained on v1+v2
  measured records; emits `InstancePlacement` trees for novel chars.
- Font-file export (Potrace / FontForge).
- Runtime ComfyUI authoring surface (interactive styling rather than
  batch).

## Operational note

Any new failure mode gets assigned to a version in this document
before it becomes a ticket. If no version subsumes it, either extend
a version or explicitly record it as won't-fix. No more "structural,
out of scope" escape hatch.
